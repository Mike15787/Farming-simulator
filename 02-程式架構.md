# 種田遊戲 — 程式架構設計

本文件記錄每個系統部分的**設計邏輯**，說明「為什麼這樣設計」與「資料如何流動」，而非僅列出檔案清單。

---

## 整體架構觀

### 技術分層

```
┌─────────────────────────────────────────┐
│        Electron 外殼（主行程）            │
│  - 開視窗、載入 index.html、檔案存取權限   │
└─────────────────────────────────────────┘
                  │ 載入
                  ▼
┌─────────────────────────────────────────┐
│      渲染行程（Renderer / 遊戲本體）      │
│  ┌─────────────────────────────────────┐ │
│  │           Phaser Game 實例           │ │
│  │  ┌────────────┐  ┌────────────────┐ │ │
│  │  │ Scene 們    │  │  全域系統 / 狀態 │ │ │
│  │  │ (戶外/室內) │  │  (GameState)    │ │ │
│  │  └────────────┘  └────────────────┘ │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

### 設計核心原則

1. **資料與表現分離**：遊戲狀態（田地、背包、天數）集中於一個 `GameState` 物件，與 Phaser 的渲染邏輯解耦。Scene 只負責「讀狀態 → 畫出來」與「接收輸入 → 改狀態」，這樣存檔、跨場景共享資料、未來換美術都不會動到核心邏輯。
2. **狀態機驅動**：田地、作物、任務都用明確的狀態列舉表示，所有變化都是「狀態轉移」，避免散落各處的布林旗標導致難以維護。
3. **事件驅動時間**：不依賴即時計時器推進世界，改由玩家行為（睡覺）觸發「換日結算」，邏輯單純、易測試、也省效能。

```
├── index.html              # 渲染行程入口
├── package.json
└── src/
    ├── main.js             # Phaser 遊戲初始化設定 + 舊存檔遷移(含 v4→v5 農場放大 tiles 重建、v5→v6 補 workers、v6→v7 補 warehouse、v7→v8 補正任務村通道地形)

    ├── scenes/

    │   └── HireScene.js    # 勞力仲介所模態面板(雇用/解雇幫工 + 面板內指派田+作物)
    ├── systems/
    │   ├── FarmSystem.js   # 田地與作物狀態機邏輯（actOnTile / canActOnTile）
    │   ├── TimeSystem.js   # 換日結算(幫工結算 + 天氣結算 + 作物生長 + 農夫生產 + 需求消耗)
    │   ├── InventorySystem.js  # 背包增刪查
    │   ├── QuestSystem.js  # 任務狀態管理
    │   ├── ShopSystem.js   # 買賣交易（買種子固定價 / 賣作物走動態價）
    │   ├── MarketSystem.js # 供需動態定價 + 市場庫存 + 每日胃納/節慶(price/appetite/demandToday/consumeDemand)
    │   ├── FarmerSystem.js # NPC 農夫每日模擬生產 + 財富排行榜
    │   ├── HireSystem.js   # 幫工雇用/解雇/指派 + 每日重用 FarmSystem 做翻土/播種/澆水/收成
    │   ├── WarehouseSystem.js # 倉庫增刪查(GameState.warehouse.items,無選取格概念)
    │   └── WeatherSystem.js # 天氣 + 天氣預報 + 天氣效果(weatherForDay/forecast/applyDailyWeather)
    └── entities/
        ├── Player.js       # 玩家自由移動、碰撞、視覺面向

```

> 互動判定原本獨立為 `InteractionSystem.js`（面前一格做什麼）。改用「自由移動 + 最近目標」模型後已移除，判定併入 `FarmSystem.canActOnTile`（找最近可動作的格子）與 `FarmScene`（掃描 + 分派）。

## 各部分設計邏輯

### 1. Electron 外殼（main.js）

**職責**：建立 BrowserWindow、載入 index.html、設定視窗大小（固定，符合地圖尺寸）。

**設計邏輯**：保持極薄。遊戲全部跑在渲染行程，主行程只做開窗。存檔走 localStorage 而非檔案系統，因此初版不需要複雜的 IPC；若日後要存成實體檔案，再透過 preload 暴露受控的 IPC 介面，不破壞現有架構。

---

### 2. GameState（單一真實來源）

**職責**：持有整個遊戲的可序列化狀態，是所有系統讀寫的唯一資料源。

**結構草案**：
```js
GameState = {
  version: 8,                  // 存檔版本(v8:農場西側任務村通道地形補正)
  currentScene: 'farm',        // 'farm' | 'house' | 'town' | 'forest'
  player: { x, y, facing },    // x,y 為「當前地圖的像素座標」(自由移動);facing 僅視覺用
  farmReturn: { x, y },        // 從室內回戶外時的落點(格子座標)
  day: 1,
  money: 0,
  tiles: [ /* 農場 20x20,每格 { terrain, tilled, crop, canopy };只存農場,城鎮/森林讀 config 靜態格 */ ],
  inventory: [ { id, qty } ],  // 同 id 堆疊;名稱/顏色由 config 的 ITEMS 查表
  selectedSlot: 0,
  quests: { tomatoQuest: 'not_started' },  // 狀態列舉,依 AREA_MAPS.quests.npcs 動態產生(buildQuests())
  market: { supply: { cropId: n } },       // 市場庫存
  farmers: [ { id, step, daysIntoCrop, money } ],  // NPC 農夫動態
  weather: { seed },           // 天氣種子(天氣/預報由 seed+day 推導)
  barrierDays: 0,              // 全域魔法結界剩餘天數
  forest: { gathered: { "x,y": 採集當天day } },  // 森林採集節點的最後採集日(重生由 day 推導)
  workers: [ { id, hired, job: { x, y, crop } | null } ],  // 幫工動態:是否受雇 + 指派的田+作物(靜態定義在 config.HIRE_DEFS)
  warehouse: { items: [ { id, qty } ] }  // 倉庫:與 inventory 同款堆疊,幫工收成自動存入,唯讀檢視
}
```

> - 玩家座標自初版「格子」改為「像素」,語意再從「農場」推廣為「當前地圖」(farm/town/forest);室內固定入口不讀此值。
> - **地圖尺寸與視口解耦**:畫布固定 480×560,地圖區是 480×480 的**相機視窗**(config `VIEW_*`),世界(如農場 20×20)可比視窗大 → 相機捲動。`MAP_W/MAP_H` 由 `FARM_MAP` 推導。
> - 載入舊存檔時由 `src/main.js` 的 `migrate()` 依 `version` 自動轉換。**v4→v5**:農場放大使 `tiles` 長度改變 → 以 `buildTiles()` 重建,把舊 15×15「兩邊 terrain 皆 soil」的格子進度複製到相同座標(既有田保留、新田為空);另補 `forest.gathered` 與缺少的 `market.supply` 鍵。**v5→v6**:補上 `workers`(`buildWorkers()`,每位 `HIRE_DEFS` 一筆,預設未雇用、無指派)。**v6→v7**:補上 `warehouse`(`buildWarehouse()`,空的物品堆疊)。**v7→v8**:`FARM_MAP` 的 `(0,10)` 字元由 `W` 改 `P`(新增任務村通道),但 `tiles` 是存檔內容、不會因為改 config 就自動更新,所以把舊檔那一格的 `terrain` 從 `'water'` 直接改成 `'path'`——這是**改動單一地形字元也需要 migration** 的例子,不是只有像 v4→v5 那種尺寸變化才需要。

**設計邏輯**：
- 之所以集中管理，是因為**存檔**只要序列化這一個物件即可；**跨場景共享**（從戶外進室內，背包與天數要延續）也只要讀同一份資料。
- 刻意設計成「純資料、可 JSON 化」——不放 Phaser 的 GameObject 參照。渲染物件由各 Scene 依狀態當場生成，狀態本身不持有任何不可序列化的東西，存檔才不會出錯。
- 田地用一維 / 二維陣列而非物件雜湊，因為格子地圖天生適合用索引 `(y * width + x)` 存取，效能與序列化都單純。

### 3. SaveManager（存讀檔）

**職責**：把 GameState 寫入 / 讀出 localStorage。

**設計邏輯**：
- `save()`：`JSON.stringify(GameState)` 存入固定 key。
- `load()`：讀出後 `JSON.parse`，若不存在則回傳預設初始狀態。
- 因為 GameState 已設計為純資料，存讀檔不需要任何特殊轉換邏輯——這是前面「狀態與表現分離」設計帶來的直接好處。
- 觸發點：TimeSystem 換日後呼叫一次 `save()`，確保進度不丟。

---

### 4. Scene 設計（Phaser Scene 系統）

採用 Phaser 內建多 Scene 機制，分三個：

#### FarmScene（戶外農場）
- 啟動時讀 GameState.tiles，把每格依地形 / 作物狀態畫成色塊。
- 相機:`mapUtils.setupWorldCamera` 設視口(上方 480×480)+ 世界邊界(20×20)+ 跟隨玩家捲動;天氣覆蓋層 `setScrollFactor(0)` 螢幕固定。
- 處理玩家在戶外的移動與互動。
- 偵測玩家碰到「門格子」→ 進室內;碰到「通道格(path)」→ `mapUtils.transitionTo` 切到城鎮。切換前先把玩家狀態寫回 GameState。

#### HouseScene（室內）
- 畫出房屋內部（地板、牆、床、出口門）,以 `VIEW_W/VIEW_H` 置中(相機不捲動,房屋小於視口)。
- 偵測碰床 → 呼叫 TimeSystem 換日。
- 偵測碰出口門 → 切回 FarmScene,玩家出現在戶外門口座標。

#### AreaScene（城鎮 / 森林,資料驅動）
- 一個類別以不同 key + mapId 註冊成 `TownScene` / `ForestScene`(見 `main.js` scene 陣列)。
- 地圖是 `config.AREA_MAPS[mapId]` 的靜態字元格(**不進 GameState.tiles**,比照 HouseScene);`solidAt` 讀字元格,顏色由 `def.colors[char]` 查 `COLORS`。相機同 FarmScene(捲動)。
- 互動骨架同 FarmScene(最近目標 + 動作按鈕):森林掃描 `gatherNodes` 找最近「已重生」節點 → 採集入袋;城鎮無節點 → 動作按鈕停用,純探索。走到通道格 → 切到相鄰地圖。

#### UIScene（疊加層）
- 用 Phaser 的 Scene 疊加能力，永遠在最上層繪製背包列、天數、對話框、動作按鈕(以 `VIEW_H` 佈局,相機不捲動)。
- 與遊戲場景分離的原因：UI 不該隨地圖場景切換而重建；獨立 Scene 可常駐,只更新顯示內容。
- 動作按鈕分派給 `Runtime.gameScene`(當前戶外場景:FarmScene 或 AreaScene)的 `doAction()`,同一顆按鈕同時驅動耕作與採集。

**設計邏輯**：場景切換時，「玩家在哪個場景、座標多少」這件事的真實來源仍是 GameState(`currentScene` + `player.x/y` 為當前地圖像素);`BootScene` 以 `config.SCENE_KEY` 把 `currentScene` 對映到場景 key。這樣即使中途存檔再讀檔，也能正確還原玩家所在場景與位置。通道切換沿用「先寫 GameState → 存檔 → `scene.start`」;dest 必為非通道格,避免立即彈回。

---

### 5. FarmSystem（田地與作物狀態機）

**職責**：管理單格田地的狀態轉移與作物生長階段。

**狀態列舉**：
```
TileCropState: EMPTY → TILLED → SEEDED → WATERED → GROWING → MATURE
```

**設計邏輯**：
- 每個動作（翻土、播種、澆水、收成）都是一個明確的轉移函式，輸入「目前狀態」回傳「新狀態」，無副作用、易測試。
- 作物生長階段（0–3）與「是否已澆水（today watered 旗標）」分開記錄。澆水只設定旗標，真正的階段推進交給 TimeSystem 在換日時統一處理——這是「事件驅動時間」的體現，避免在 update loop 裡每幀檢查生長。
- 為什麼澆水與生長分離：種田遊戲的核心節奏是「今天照顧 → 明天看結果」，把澆水（即時動作）與生長（換日結算）解耦，才能正確表達這個節奏。

---

### 6. TimeSystem（換日結算）

**職責**：處理「進入下一天」時的所有結算。

**換日流程**：
```
1. day += 1
2. 遍歷所有田地格子：
     若 state == WATERED 且尚未到 MATURE
        → 生長階段 +1，重設「今日已澆水」旗標
     若仍為 SEEDED（當天沒澆水）→ 停滯，不前進
3. 推進相關任務 / 事件（如有）
4. 呼叫 SaveManager.save()
```

**設計邏輯**：把所有「跨天才會發生的變化」集中在這一個函式，世界的時間推進就只有一個入口，邏輯清楚、不會有遺漏的狀態沒更新。日後要加日夜循環或季節，也只需擴充這裡。

---

### 7. InventorySystem（背包）

**職責**：對 GameState.inventory 做增、刪、查、計數。

**設計邏輯**：
- 提供 `addItem(id, qty)`、`removeItem(id, qty)`、`countItem(id)` 等純函式介面。
- 同類物品堆疊（同 id 累加數量），而非每個佔一格，符合種田遊戲習慣也省 UI 空間。
- QuestSystem 需要「玩家有幾個番茄」時，呼叫 `countItem` 即可，不直接碰 inventory 陣列——保持單一存取入口，避免邏輯散落。

---

### 8. QuestSystem（任務狀態,依 questId 分派多任務）

**職責**：管理任務狀態列舉與條件檢查,並依 questId 分派給對應的任務。

**狀態列舉**（每個任務各自一份,狀態存在 `GameState.quests[questId]`）：
```
not_started → in_progress → completed
```

**設計邏輯**：
- **`HANDLERS` 依 questId 查表**:每個任務是 `HANDLERS` 裡一個具名項目,提供
  `talk(state)`(對話內容 + 交付副作用,回傳 `{lines, onEnd}`)與 `hasMarker(state)`(NPC 頭上「!」
  是否顯示)。`QuestSystem.talk(state, questId)` / `QuestSystem.hasMarker(state, questId)` 只是查表
  轉發,呼叫端(`AreaScene`)不需要知道任務內部邏輯。
- 交付檢查：各任務的 `talk` 內部呼叫 `InventorySystem.countItem` 判斷是否達標 → 達標則扣除物品、
  給獎勵、狀態轉 `completed`。
- **哪些任務存在,由地圖資料決定**:`config.AREA_MAPS.quests.npcs` 這份 NPC 座標清單本身就是「有哪些
  任務」的唯一真實來源(見第 13 節),`GameState.buildQuests()` 依此自動產生每個任務的預設狀態,不必
  另外維護一份任務 id 清單。
- 用狀態列舉而非散布的布林值,是為了讓「NPC 該說什麼」變成單純的狀態查表;用 questId 查表分派,是為了
  讓「支線任務」從一份構想變成可以無限疊加的擴充點——新增任務只要加一個 `HANDLERS` 項目 + 一筆 npc
  座標,不必碰互動判定或場景程式碼。

---

### 9. 互動判定（最近目標 + 動作按鈕）

> 初版設計為獨立的 `InteractionSystem`,依玩家「面前一格」分派動作。改用自由移動後,「面前一格」不再可靠,改為**對玩家附近最近的可動作目標做事**,判定併入 `FarmSystem` 與 `FarmScene`,`InteractionSystem.js` 已移除。

**職責**：玩家按動作鍵 / 點動作按鈕時,決定要對哪一格、做什麼動作。

**判定流程**：
```
每幀(FarmScene.update):
  1. 掃描所有土格,對每格呼叫 FarmSystem.canActOnTile(state, x, y, 選取物品)
     → 回傳 'till'|'plant'|'water'|'harvest'|null(不改狀態,純查詢)
  2. 取 REACH(可及距離)內、離玩家最近且非 null 的那格當「目標」,並高亮
  3. 把目標的動作名稱寫進 runtime(供 UIScene 動作按鈕顯示與啟用)

按下動作鍵 / 點動作按鈕(FarmScene.doAction):
  - 對話中            → 推進對話(交給 UIScene)
  - 否則有目標         → 交給 FarmSystem.actOnTile 執行該格動作(或商店/市場/倉庫等面板、農夫對話)
  走到「門」格 → 由 FarmScene 直接觸發場景切換(碰觸,不需按鍵)
```

> 任務 NPC 的對話不在 FarmScene——農場已無任何 NPC(見第 13 節任務村)。`AreaScene`(城鎮/森林/任務村
> 共用)有自己一份幾乎相同的判定骨架:掃描 `gatherNodes`/`buildings`/`npcs` 三種可選資料當候選,
> `doAction` 對 `npc` 分支交給 `QuestSystem.talk` 開啟對話,其餘邏輯與此處一致。

**設計邏輯**：
- `canActOnTile` 是 `actOnTile` 的「唯讀鏡像」:前者只查詢、後者才改狀態,兩者判定必須一致。把「找最近目標」與「實際執行」分開,讓 UI 能預先顯示將發生的動作、畫面能高亮目標,操作更直覺。
- 動作按鈕屬於 UI(UIScene),但不持有遊戲邏輯:它只讀 `runtime` 的標籤/可用狀態來繪製,點擊時回呼 `FarmScene.doAction()`(或 `AreaScene.doAction()`,依 `Runtime.gameScene` 決定)。鍵盤(空白/E)與滑鼠按鈕是同一條路徑的兩個入口。
- 未來加新互動物件,在對應場景的目標掃描 / 分派多一個分支即可(城鎮/森林/任務村已示範:`gatherNodes`/`buildings`/`npcs` 三種資料驅動清單都是同一個「掃描 → 取最近 → 分派」模式的重複套用)。

### 10. Player / NPC（entities）

**Player 設計邏輯**（已改為自由移動）：
- 負責處理鍵盤輸入轉成移動。位置以**像素 (x, y)** 表示,每幀依輸入向量移動;對角線會正規化,速度不因斜走而變快。
- **加速走動**:按住 Shift(`this.cursors.shift`,Phaser `createCursorKeys()` 內建就有,不必另外註冊按鍵)時,當幀速度乘上 `PLAYER_SPRINT_MULT`;純粹是移動輸入層的調整,套用在正規化之後的向量上,不影響碰撞/互動判定,對角線加速時也不會變得比直線加速更快。
- **碰撞**採「分軸 + 方框四角取樣」:先試 X 再試 Y,撞到實心格就退回該軸 → 可沿牆滑行、卡角不穿牆。某格是否實心由場景提供的 `solidAt(tileX, tileY)` 判定(水/牆/界外/NPC/建物,依場景而定)。
- `facing` 仍維護,但只剩**視覺用途**(角色朝向的小方塊),不再用於互動判定。
- 玩家的「真實座標」(像素)最終要同步回 GameState(場景切換 / 存檔前)。
- 室內(HouseScene)與戶外共用同一個 Player,差別只在傳入的 `origin` 位移與 `solidAt`。

**NPC 設計邏輯**(已泛化為可放多個)：
- constructor 吃一個 `def`(`{x, y, color?, name?}`),不再寫死單一座標——`AreaScene` 可依
  `config.AREA_MAPS[mapId].npcs` 這份清單實例化任意數量的 NPC(見第 13 節)。
- 持有對話資料（依任務狀態分組的台詞,實際由 `QuestSystem` 依 questId 提供）與所在座標、外觀。
- 本身不含任務邏輯，只在被互動時把控制權交給 QuestSystem，保持「資料」與「規則」分離。

### 11. 商店 / 市場（ShopSystem + ShopScene）

**職責**：商店買種子、市場賣作物,讓玩家透過經濟循環(賣作物 → 買新種子 → 種植收成)累積金錢與作物種類。

**ShopSystem(邏輯,無表現)**：
- `buy(state, id, qty)`:錢夠才扣款並 `InventorySystem.addItem`。
- `sell(state, id, qty)`:可賣才扣作物並加金錢(`qty` 為上限,有多少賣多少)。
- 價格來自 `config` 的 `ITEMS`:種子有 `buy`、作物有 `sell`(>0 才可賣);商店上架清單為 `SHOP_STOCK`。
- **多作物可擴充性**:新增作物只要在 `ITEMS` 加 `xxx_seed` 與 `xxx` 兩筆(含價格/顏色),必要時加進 `SHOP_STOCK`,生長與買賣都自動適用,不需改邏輯。

**ShopScene(表現,模態)**：
- 單一場景兩種模式(`mode: 'shop' | 'market'`),由 `FarmScene.openShop()` 以 `scene.launch` 疊在最上層;面板蓋住地圖區、保留底部 UI 列。
- 只負責畫面與點擊命中判定,交易呼叫 `ShopSystem`,完成後重建列並 `SaveManager.save`。
- 開啟期間 `Runtime.shopActive = true`:`FarmScene` 暫停、`UIScene` 不吃點擊,輸入由本面板獨佔。Esc / ✕ / 點 UI 列關閉。

**設計邏輯**:延續「資料與表現分離」——買賣規則在 ShopSystem(可單元測試),面板只是另一個「讀狀態→畫、點擊→呼叫系統」的 Scene。商店/市場在世界中是兩個色塊物件,沿用「最近目標」模型納入互動候選(見第 9 節),不需要新的輸入機制。

### 12. 多農夫競爭與動態市場（MarketSystem + FarmerSystem)

**職責**:讓市場價格隨供需浮動、NPC 農夫每天生產競爭,並以財富排行榜呈現「傳說中的農夫」。

**狀態(GameState,v4)**
- `market.supply[cropId]` — 各作物會累積的市場庫存(初始 = 需求量 → 起手價 = 基準價)。
- `farmers[]` — NPC 農夫的「動態狀態」`{ id, step, daysIntoCrop, money }`。靜態定義(名字/顏色/位置/種植流程/天數/產量)在 `config.FARMER_DEFS`,動態與靜態分離,存檔只存會變的部分。
- `weather.seed` — 天氣種子(單一整數)。天氣/預報全部由 `seed + 日期` 推導,不存每日天氣。
- `barrierDays` — 全域魔法結界剩餘天數(>0 期間全農場免疫颱風)。
- `tile.canopy` — 該格是否搭了棚子(防颱)。日期不存(由 `config.START_DATE + day` 推導)。
- **v3→v4 遷移**:`migrate()` 補 `weather.seed`(`buildWeather`)與 `barrierDays=0`;`tile.canopy` 缺值靠讀取端容錯。

**MarketSystem(動態定價,純邏輯)**
- `price(state, crop)`:`base*(1+factor)*appetite`,`factor=clamp(-K*(supply/demand-1), -MAX_DROP, +MAX_RISE)`,再夾地板價。常數在 `config.MARKET`。
- `appetite(state, crop)`:今日市場胃納 = `dailyNoise(day,crop) × 節慶倍數`。`dailyNoise` 是以 (天數+作物) 為種子的確定性偽隨機(FNV-1a),回傳約 `[1-DEMAND_NOISE, 1+DEMAND_NOISE]`;節慶當天對應作物再 ×`mult`。價格與需求消耗都乘它。
- `addSupply(state, crop, n)`:供給 +n,夾在 `[0, demand*SUPPLY_CAP_MULT]`。
- `demandToday(state, crop)`:當日需求 `round(demand*appetite*(1 - ELASTICITY*factor))`(非負)。胃納高(波動/節慶)吃更多;價低(factor<0)吃更多、價高吃更少。
- `consumeDemand(state)`:每作物 `supply -= demandToday`(不為負),換日呼叫。
- `trend(state, crop)`:比較當前價 vs 基準價回 `up/down/flat`,自動涵蓋供給、每日波動、節慶三者。
- **為何存量模型**:供給累積成庫存、被需求逐日消耗,才會有「囤貨/搶種壓低價、之後慢慢回升」的經濟慣性。玩家賣出逐個 +1 供給,體現「洪水般供給壓低價格」的邊際遞減。
- **為何需求彈性**:固定 demand 面對高產玩家一直倒貨時追不上,供給會頂在上限、價格黏在地板。彈性需求讓價低時市場多吃一些,倒貨造成的低價回穩更快,同時仍懲罰單一作物洗版(倒得比市場當日胃納多還是拿地板價)。
- **為何胃納(每日波動 + 節慶)由天數推導**:`appetite` 只依 `day` 計算(`dailyNoise` 確定性 + `config.festivalFor(day)` 純函式),所以不新增任何存檔狀態、**不需 bump SAVE_VERSION 或 migrate**,且同一天內每次呼叫值固定(價格不閃爍)。節慶讓玩家「平日囤貨、節慶當天大量賣出」(需求暴增 → 一次倒貨也能維持高價)。

**FarmerSystem(模擬農場 + 排行榜,純邏輯)**
- `stepDay(state)`:每位農夫 `daysIntoCrop++`;到 `growDays` → 收成:`MarketSystem.addSupply(crop, yield)`、`money += yield*price`、輪到 routine 下一作物。由 `TimeSystem.nextDay` 在作物生長後呼叫,需求消耗之前。
- `ranking(state)`:玩家 + 所有農夫依 money 由高到低,供排行榜面板與農夫對話使用。

**WeatherSystem(天氣 / 預報 / 效果,純邏輯)**
- `weatherForDay(state, absDay)`:某天真實天氣 `{ am, pm, typhoon }`。以 `hash(seed, absDay, seg)`(FNV-1a,同 MarketSystem 手法)依季節加權表(`config.SEASON_WEATHER`)抽上午/下午;颱風只在夏季以 `TYPHOON_CHANCE` 判定;颱風後 `TYPHOON_TAIL_DAYS` 天強制降雨(颱風尾)。
- `today(state)` / `dayAutoWaters(w)`:今日天氣 / 當天是否會自動澆水。
- `forecast(state)`:未來 `HORIZON` 天;準確率 100% 起每天 −`DECAY`,以固定 reveal 門檻做混淆(越近越揭露真值);颱風於 `WARN_DAYS` 內才顯示警報,更遠藏成大雨。
- `nextTyphoonWithin(state, days)`:最近颱風(供 HUD 警報)。`isProtected(state, i)`:該格 `canopy` 或全域 `barrierDays>0`。
- `applyDailyWeather(state, endedDay)`:颱風的**非作物**效果 —— 市場供給 ×`SUPPLY_SHOCK`、棚子依 `CANOPY_BREAK_CHANCE` 吹壞、`barrierDays` 遞減。作物的自動澆水/生長/農損由 `TimeSystem` 田地結算迴圈處理(順序:先田地、再本函式)。
- **為何天氣由天數推導**:只存 `weather.seed`,天氣/預報皆為 `seed + day` 的純函式 → 未來可算(預報)、跨讀檔穩定、每局不同、同一天固定;調 `config` 天氣表不需 migration。

**接線(天氣)**:`TimeSystem.nextDay` 先算 endedDay 天氣 → 自動澆水 + 颱風農損(田地迴圈)→ `WeatherSystem.applyDailyWeather`;`FarmScene` 加天氣覆蓋層、`WEATHER_POS`「報」布告欄、搭棚(`canopy_kit` → `FarmSystem` 的 `canopy` op)與棚子繪製;`UIScene` 顯示日期 + 天氣 emoji + 颱風警報徽章;`ShopScene` 新增 `forecast` 模式(14 天預報 + 施放結界)。

**接線**:`ShopSystem.sell` 改用 `MarketSystem.price` 並逐個 `addSupply`;`TimeSystem.nextDay` 串 `FarmerSystem.stepDay → MarketSystem.consumeDemand`;`FarmScene` 把 3 位農夫與「榜」色塊納入最近目標(對話 / 開排行榜面板);`ShopScene` 新增 `rank` 模式渲染排行榜、`market` 模式顯示動態價與漲跌。**新增作物零改動**:在 `ITEMS` 補上 `sell`(基準價)與 `demand`,定價/供給/市場自動適用。

### 13. 多地圖 + 捲動相機 + 森林採集 + 任務村（AreaScene + mapUtils）

**職責**:在原本的農場/室內之外,新增城鎮、森林、任務村三張可探索地圖,並讓單張地圖可比一屏大
(相機捲動);森林可採集資源接進市場經濟,任務村集中放置所有任務 NPC。

**視口 / 地圖尺寸解耦**
- 畫布固定 480×560;地圖區是 480×480 的**相機視窗**(`config.VIEW_*`),與地圖世界大小分離。`GAME_W/H` 由 `VIEW_*` 推導,`MAP_W/H` 由 `FARM_MAP` 推導(農場現 20×20)。
- `mapUtils.setupWorldCamera(scene, worldW, worldH, target)`:設主相機 viewport=上方視窗、bounds=世界、`startFollow(玩家)`。FarmScene 與 AreaScene 共用。
- 螢幕固定層(天氣覆蓋、UI、商店面板)用 `setScrollFactor(0)` 或獨立不捲動的 Scene,並以 `VIEW_H`(非 `MAP_H*TILE`)佈局。

**多地圖(資料驅動)**
- 城鎮/森林/任務村是 `config.AREA_MAPS[mapId]` 的靜態字元格(`{ layout, colors, solid, gatherNodes?, buildings?, npcs? }`),由單一 `AreaScene` 類別以三個 key(`TownScene`/`ForestScene`/`QuestScene`)註冊。只有農場進 `GameState.tiles`(唯一可耕作、需持久);其餘地圖每次由 config 重建(比照 HouseScene)。`buildings?`(如城鎮的勞力仲介所,第 14 節)與 `npcs?`(如任務村的任務 NPC,見下方)都是同一個「資料驅動、AreaScene 掃描最近目標」模式的重複套用,寫法比照最早的 `gatherNodes`。

**通道切換(觸碰)**
- `config.PORTALS[mapId]` 列出 `{ x, y, to, dest }`;各戶外場景每幀檢查玩家所在格,命中就 `mapUtils.transitionTo(scene, to, dest)`(寫 `GameState.player=dest 像素 + currentScene` → 存檔 → `scene.start(SCENE_KEY[to])`)。**dest 必為非通道格**否則會立即彈回。拓撲:任務村 ⇄ 農場 ⇄ 城鎮 ⇄ 森林(農場東西各一個通道口,城鎮是城鎮/森林側的樞紐)。

**森林採集**
- 沿用「最近目標」模型:`AreaScene.computeAction` 掃 `gatherNodes`(`{ x, y, item, respawnDays }`)找 REACH 內最近的**已重生**節點;`doAction` → `InventorySystem.addItem`(同收成路徑)+ 記 `GameState.forest.gathered["x,y"]=day` + 存檔 + 浮出「+物品」文字。
- **重生由天數推導**:節點在 `day - gathered >= respawnDays` 時再次可採,只存「最後採集日」一個數字(比照天氣由 day 推導的精神)。
- 採集物(`berry`/`mushroom`)是一般 `ITEMS`(有 `sell`+`demand`,無 `_seed`),`buildMarket` 自動建供給 → 直接以動態市價在市場販售,零特例。

**任務村(多 NPC + 多任務基礎建設)**
- **NPC 資料驅動**:`def.npcs`(`{ id, x, y, color, name? }`)是繼 `gatherNodes`/`buildings` 之後第三種
  可選的「最近目標」清單。`entities/NPC.js` 已從「建構子寫死讀農場那個 `NPC_POS`」泛化為「吃一個
  `def` 物件」,`AreaScene.create()` 就能依 `def.npcs` 實例化任意數量。`solidAt`/`computeAction`(候選
  `{kind:'npc', x, y, npcId}`)/`doAction`(呼叫 `QuestSystem.talk(state, npcId)` → `ui.startDialog`)
  都是同一套「掃描候選 → 取最近 → 分派」模式,`update()` 另外每幀把每個 NPC 的「!」標記依
  `QuestSystem.hasMarker(state, npcId)` 更新。**農場已不再有任何 NPC**——原本唯一的番茄任務 NPC 搬到
  任務村。
- **為何要一張獨立地圖**:把所有任務 NPC 集中在一處,是為了讓開發者測試多條任務線路時不必在
  農場/城鎮/森林之間往返奔波,純粹是開發效率的考量,不影響其他系統。
- **任務 id 的唯一真實來源**:`GameState.buildQuests()` 依 `AREA_MAPS.quests.npcs` 的 id 產生每個任務的
  預設狀態(見第 8 節),新增任務線路只要在 `npcs` 加一筆座標 + 在 `QuestSystem.HANDLERS` 加一個同名
  handler,兩處都不需要改動 `AreaScene`/`NPC.js`。`quests` 欄位本身**不需要 bump SAVE_VERSION**:
  `state.quests.tomatoQuest` 這個既有欄位形狀完全沒變,只是預設值的產生方式從寫死改成查表推導。
- **但 `FARM_MAP` 改字元(新增西側通道)還是需要 migration**:一開始誤以為整條功能都不用動存檔版本,
  忽略了 `tiles` 是「開新局那一刻用 `FARM_MAP` 算好、之後就是純存檔內容」——改 config 的字元不會讓
  已存在的存檔跟著變,所以舊檔那格的 `terrain` 仍是 `'water'`,玩家實際上永遠走不到新通道(見第 15
  節下方 v7→v8)。這是本節唯一需要 bump 到 v8 的原因,和 quests 欄位本身無關。

**設計邏輯**:延續「資料與表現分離」與「盡量由 day 推導、少存檔」。地圖、通道、採集點、建物、NPC 全是
config 資料;新增一張地圖 = 加一筆 `AREA_MAPS` + `PORTALS`;新增採集物 = 一筆 `ITEMS` + 一個
`gatherNodes`;新增任務線路 = 一筆 `npcs` + 一個 `QuestSystem.HANDLERS` 項目。相機/切換/採集/對話邏輯
與具體地圖無關,可重用。

### 14. 雇用系統(HireSystem + HireScene)

**職責**:城鎮新增勞力仲介所,讓玩家雇用 NPC 幫工,並在同一個面板內指派工作(選田 + 選作物);
受雇者每天自動對指派的田做翻土/播種/澆水/收成,工作範圍先限定農業。

**通用建物點(推廣 gatherNodes)**
- `AREA_MAPS.town.buildings`(`{ id, x, y, kind, label, colorKey }`)把原本只有森林在用的「靜態可互動點」
  (`gatherNodes`)推廣成任何地圖都能選用的機制。`AreaScene.computeAction` 把 `buildings` 當成 `gatherNodes`
  以外的另一批最近目標候選;`doAction` 依 `kind` 查一個 `BUILDING_SCENE` 對照表(`{agency:'HireScene'}`)
  找場景 key,以 `scene.launch()+bringToTop()` 開啟(手法同 `FarmScene.openShop()`);`drawBuildings()` 於
  `create()` 畫一次(靜態位置,不隨互動重繪,比照 `FarmScene.drawWorldObjects` 的色塊+標籤畫法)。

**狀態(GameState,v6)**
- `workers[]` — 幫工的「動態狀態」`{ id, hired, job:{x,y,crop}|null }`。靜態定義(名字/顏色/雇用費/日薪)
  在 `config.HIRE_DEFS`,動態與靜態分離,存檔只存會變的部分(比照 `FarmerSystem` 的 `FARMER_DEFS`/`farmers` 分工)。
- **v5→v6 遷移**:`migrate()` 補 `workers`(`buildWorkers()`)。

**HireSystem(雇用/指派/每日結算,純邏輯)**
- `hire(state, id)` / `dismiss(state, id)`:雇用扣一次性 `hireCost`;解雇清空 `hired`+`job`,不退費。
- `assign(state, id, x, y, crop)`:目標須為農場可耕土格且未被「別的」受雇員工佔用,否則回傳
  `{ok:false, reason:'tile_taken'}`;`unassign(state, id)` 取消指派(仍受雇)。
- `stepDay(state)`:**受雇者是真的在玩家自己的農場格子上工作,不是像 FarmerSystem 那樣背景模擬的抽象
  產出** —— 直接重用 `FarmSystem.canActOnTile`/`actOnTile`,以一個合成的 `selected = {id: crop+'_seed',
  qty: 即時查詢的背包庫存}` 傳入,對指派格做「今天的一步」(翻土/播種/澆水/收成),種子從玩家真實背包扣,
  不必另寫一套平行的生長邏輯。**每位員工的迴圈內都即時 `countItem`,不可迴圈外快取**,否則兩位員工同天
  搶種同一種子、庫存只剩 1 顆時會重複判定「有貨」而多扣。**收成不進玩家背包**:`actOnTile` 的收成分支
  只認得 `state.inventory`,所以先讓它照常加進去,`stepDay` 緊接著呼叫 `InventorySystem.removeItem` +
  `WarehouseSystem.addItem`(見第 15 節)把剛收的那一份搬到倉庫——幫工的收成只會出現在倉庫,不會混進
  玩家隨身背包。同時扣每位受雇者的日薪(`state.money >= wage` 才扣,錢不夠就先欠著、不強制解雇)。

**接線(換日順序)**:`TimeSystem.nextDay` 在**田地生長迴圈之前**先呼叫 `HireSystem.stepDay(state)` ——
這樣幫工今天的澆水動作,會被緊接著的生長迴圈用同一次呼叫正確結算(與玩家自己白天澆水、晚上結算的
節奏一致);颱風農損迴圈同理正確套用在幫工剛種下/剛澆水的格子上。

**HireScene(表現,模態,兩個子畫面)**
- `list`(預設):`HIRE_DEFS` 逐一列出,未雇用顯示雇用鈕;已雇用顯示日薪、目前工作摘要、「指派工作」
  (切到 assign 畫面)與「解雇」鈕。
- `assign`(針對某位員工):在面板內畫一張縮小版農場全圖(`MAP_W×MAP_H`,約 11px/格,獨立模態場景不掛
  世界相機,座標命中判定與 `ShopScene.onPointer` 既有的矩形命中法一致)——非可耕地淡灰不可點,已被
  別的員工佔用的可耕地疊該員工顏色;點擊設目前選取格,下方一列作物色塊(`config.plantableCropIds()`,
  即所有有 `_seed` 對應的作物)點擊設選取作物;「確認指派」(需同時選格+選作物才可按)呼叫
  `HireSystem.assign`,「取消指派」呼叫 `unassign`,「返回」回 list 畫面。**指派全程在面板內完成,不需
  離開去農場找那塊田。**

**設計邏輯**:延續「資料與表現分離」——雇用/指派/每日結算規則在 `HireSystem`(可單元測試),`HireScene`
只是另一個「讀狀態→畫、點擊→呼叫系統」的模態 Scene,沿用 `ShopScene` 的既有慣例(`Runtime.shopActive`
暫停世界、Esc/✕/點面板外關閉)。幫工重用 `FarmSystem` 而非另寫一套,是因為它們操作的是**同一份**
`GameState.tiles`——玩家與幫工可以同時「共同經營」同一塊田,狀態機只有一套,不會有兩邊邏輯不同步的風險。
`FarmScene.drawField()` 額外在被指派的格子角落畫一個該員工顏色的小標記,純視覺,不影響互動判定。

### 15. 倉庫(WarehouseSystem)

**職責**:讓幫工(第 14 節)自動收成的作物有個歸屬,和玩家隨身背包分開放;提供一個唯讀面板讓玩家
看倉庫裡有什麼。刻意不做搬運/存取操作,維持最小可行範圍。

**建物(沿用最近目標模型)**
- `config.WAREHOUSE_POS`(房子東側的草地)是農場地圖上的一個色塊建物,寫法與 `SHOP_POS`/`MARKET_POS`
  完全一樣:`FarmScene.solidAt` 視為實心、`computeAction` 的 `consider('warehouse', WAREHOUSE_POS)` 納入
  最近目標候選、`drawWorldObjects` 畫色塊+「倉」字標籤、`doAction` 對應分支呼叫
  `this.openShop('warehouse')`——沒有新增任何互動機制,單純多一個候選建物。

**狀態(GameState,v7)**
- `warehouse.items`:與 `inventory` 同款 `{id,qty}` 堆疊。
- **v6→v7 遷移**:`migrate()` 補 `warehouse`(`buildWarehouse()`,空陣列)。

**WarehouseSystem(增刪查,純邏輯)**
- `addItem`/`removeItem`/`countItem`,邏輯與 `InventorySystem` 幾乎一致,但**不處理 `selectedSlot`**——
  那是隨身背包「目前選取格」的 UI 概念,倉庫沒有選取格(唯讀檢視),所以獨立成一個小檔案而非讓
  `InventorySystem` 多接一個「目標陣列」參數,避免核心背包邏輯因為一個單向用途被複雜化。
- 目前唯一的寫入者是 `HireSystem.stepDay` 的收成搬運(見第 14 節);玩家親自收成仍照舊進 `inventory`。

**ShopScene `warehouse` 模式(表現,唯讀)**
- `computeRows()` 回傳 `state.warehouse.items` 映射成色塊+名稱+數量;`buildRows()` 比照 `rank` 模式的
  唯讀列寫法(色塊+名稱是所有列共用的部分,`warehouse` 分支只加一個靠右的 `×數量` 文字,不加按鈕)。
  空倉庫時顯示「倉庫目前是空的」(比照市場沒有可賣作物的提示)。

**設計邏輯**:倉庫是「幫工自動化」帶來的一個小配套——沒有它,幫工收成就得混進玩家隨身背包,擠占本來
給種子/工具用的格子。用一個獨立、極簡的第二堆疊 + 唯讀面板解決,不做存取操作是因為目前沒有需求,
硬做反而是本檔案開頭就強調要避免的「超前設計」。

## 資料流總結

```
玩家輸入(鍵盤移動 / 動作鍵 / 滑鼠點動作按鈕)
   │
   ▼
FarmScene / AreaScene（自由移動 + 掃描最近可動作目標）
   │
   ├─→ FarmSystem.actOnTile ─────────┐   (鋤地/播種/澆水/收割/搭棚,農夫對話查看狀態)
   ├─→ QuestSystem.talk(任務村 NPC) ─┤   (AreaScene:靠近任務 NPC,依 questId 分派)
   ├─→ ShopScene → ShopSystem ───────┤   (商店買 / 市場賣[動態價] / 榜看排名 / 報看預報+施放結界 / 倉庫唯讀)
   ├─→ HireScene → HireSystem ───────┤   (城鎮仲介所:雇用/解雇幫工、面板內指派田+作物)
   ├─→ AreaScene 採集 ───────────────┤   (森林 gatherNodes)
   └─→ Scene 切換 ───────────────────┤   (走到門格 / 通道格)
                                     ▼
                               修改 GameState（單一真實來源)
                                     ▲
   睡覺換日 TimeSystem.nextDay ──────┤
     HireSystem.stepDay(幫工對指派田做今天的一步:翻土/播種/澆水/收成 + 扣日薪;收成經
       InventorySystem→WarehouseSystem 搬進倉庫,不進玩家背包)
       → 天氣結算(自動澆水 / 颱風農損 / WeatherSystem.applyDailyWeather:供給衝擊·吹棚·結界遞減)
       → 作物生長 → FarmerSystem.stepDay(農夫產出灌供給) → MarketSystem.consumeDemand(當日胃納×彈性需求消耗)
     (天氣/預報由 weather.seed + 日期推導,見 WeatherSystem;價格波動/節慶由 day 推導,見 MarketSystem.appetite)
                                     │
        ┌────────────────────┬───────┴────────────┐
        ▼                    ▼                     ▼
   各 Scene 重繪        UIScene 更新           SaveManager 存檔
   （讀狀態畫色塊 +      （背包/天數/金錢 +
    高亮目標格）          動作按鈕標籤）
```

**一句話總結架構精神**：所有規則改的是 GameState，所有畫面讀的是 GameState，存檔存的是 GameState。把這條線守住，遊戲就好維護、好擴充、好除錯。

## 建議實作順序

1. Electron + Phaser 跑起一個空白視窗與一個 Scene。
2. config + GameState 初始結構 + 戶外地圖色塊渲染。
3. Player 自由移動與碰撞(solidAt)。
4. FarmSystem(actOnTile / canActOnTile)+ 最近目標掃描:鋤地→播種→澆水→收割跑通（先不換日，手動測）。
5. TimeSystem + HouseScene + 睡覺換日，作物會長大。
6. InventorySystem + UIScene 背包顯示。
7. NPC + QuestSystem：一個完整任務循環。
8. SaveManager：存讀檔。
9. ShopSystem + ShopScene：商店買種子 / 市場賣作物(多作物 + 經濟循環)。
10. MarketSystem + FarmerSystem:供需動態定價 + NPC 農夫每日生產 + 財富排行榜。
11. 天氣 / 日期系統(WeatherSystem:季節天氣 + 預報 + 防災)。
12. 多地圖 + 捲動相機(視口/地圖解耦 → 農場放大 + 兩塊新田 → AreaScene 城鎮/森林 + 通道切換 → 森林採集)。
13. 雇用系統(HireSystem + HireScene:城鎮仲介所建物 → 雇用/解雇 → 面板內指派田+作物 → TimeSystem 換日重用 FarmSystem 自動翻土播種澆水收成)。
14. 倉庫(WarehouseSystem + ShopScene warehouse 模式:房子旁建物 → 幫工收成自動搬入 → 唯讀面板檢視)。
15. 任務村(NPC/QuestSystem 泛化為多 NPC、依 questId 分派 → 農場西側新通道 → AreaScene npcs 機制 → 番茄任務 NPC 搬遷)。
16. 收尾打磨（互動提示、對話框體驗、平滑移動）。
